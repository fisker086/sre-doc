# 主从复制延时问题的原因分析及处理

## 一、什么是主从延时？

```mysql
主库发生了操作，从库“很久”才跟上发来

影响因素：
    网络 
    主从硬件差异较大
    版本差异
    参数因素
```



## 二、影响因素

### 1、外在因素

```mysql
网络 
主从硬件差异较大
版本差异
参数因素
```



### 2、主库

```mysql
(1) 二进制日志写入不及时
select @@sync_binlog;
(2) CR的主从复制中,binlog_dump线程,事件为单元,串行传送二进制日志(5.6 5.5)

1. 主库并发事务量大,主库可以并行,传送时是串行
2. 主库发生了大事务,由于是串行传送,会产生阻塞后续的事务.

解决方案:
1. 5.6 开始,开启GTID,实现了GC(group commit)机制,可以并行传输日志给从库IO
2. 5.7 开始,不开启GTID,会自动维护匿名的GTID,也能实现GC,我们建议还是认为开启GTID
3. 大事务拆成多个小事务,可以有效的减少主从延时.
```



### 3、从库

```mysql
SQL线程导致的主从延时
在CR复制情况下: 从库默认情况下只有一个SQL,只能串行回放事务SQL
1. 主库如果并发事务量较大,从库只能串行回放
2. 主库发生了大事务,会阻塞后续的所有的事务的运行

解决方案:
1. 5.6 版本开启GTID之后,加入了SQL多线程的特性,但是只能针对不同库(database)下的事务进行并发回放.
2. 5.7 版本开始GTID之后,在SQL方面,提供了基于逻辑时钟(logical_clock),binlog加入了seq_no机制,
真正实现了基于事务级别的并发回放,这种技术我们把它称之为MTS(enhanced multi-threaded slave).
3. 大事务拆成多个小事务,可以有效的减少主从延时.
[https://dev.mysql.com/worklog/task/?id=6314]
```







## 三、主从延时怎么监控？

### 1、粗略评估

```mysql
mysql> show slave status \G;
Seconds_Behind_Master: 0	
#0：指slave的两个线程的时间差值。只能得出：有无延时情况，0也不一定没有延迟
```



### 2、评估主从延时的更加精确的指标：日质量评估

```mysql
评估主从延时的更加精确的指标是：延时了多少日志量
同一时刻：主库执行的日质量和从库执行的日质量的对比

日志量：主库binlog位置点 对比 从库relay执行的位置点
```



### 3、如何计算延时的日志量？

```mysql
show master status;
cat /service/mysql/data/relay-log.info
```



### 4、主从复制延时的原因

#### 1）主库原因

````mysql
外部因素：
	1.网速、硬件配置、主库业务繁忙、从库太多
	
内部因素：
1、二进制日志更新不及时。sync_binlog=1

2、 5.7之前的版本，没有开GTID之前，主库可以并发事务，但是dump传输时是串行的，所以会导致，多事务、大事务时会出现比较严重的延时。
	5.6+版本，手工开启GTID，事务在主从的全局范围内就有了唯一性标识。
	5.7+版本，无需手工开启，系统会自动生成匿名的GTID信息
	有了GTID之后，就可以实现并发传输binlog
	但是，及时有这么多的优秀特性，我们依然需要尽可能的减少大事务以及锁的征用问题。
	
	
主库业务繁忙怎么办？
	1.拆分业务（分布式）：组件分离	垂直	水平
	2.大事务拆分。比如1000w行的业务，分20次执行
````

**如何判断是主库传输不及时？**

````mysql
1.seconds_behind_master
2.主库：show master status;
  从库：show slave status;
````





#### 2）从库原因

```mysql
外部因素：
	网络、配置低、参数设定。

内部因素：
	
1、IO线程
	1.写relay-log ———》自身IO性能

2、SQL线程
	1.非GTID情况下默认单线程串行回放事务。
		解决：
			1.开启GTID
			2.串行改并行
				5.6+ GTID：database级别，基于库级别SQL线程并发
				5.7+ GTID：logic_clock逻辑时钟。保证了同库级别下的事务顺序问题，所以可以理解为基于事务级别的并发回放。MTS
				
	PS：及时有以上的自带的优化机制，我们还是要注意对于大事务、及锁的征用问题。
```



**总结：mysql现有的新版，5.7+在主从延时方面已经解决掉遗留的很多问题，所以我们着重解决人的问题，大事务和锁是需要我们额外关注的**
