# 索引的基本管理

## 一、什么时候创建索引?

```mysql
1.按照业务的需求创建合适的索引。
2.并不是将所有列都建立索引。不是索引越多越好。
3.将索引建立在，经常 where group by join on 的条件

为什么不能随便建索引?
1.如果冗余索引过多，表的数据变化，很可能导致索引频繁更新。会阻塞很多正常的业务更新请求。
2.索引过多，会导致优化器选择出现偏差
```

## 三、执行计划分析

### 1、什么是执行计划

```mysql
(1)
获取到的是优化器选择完成的,他认为代价最小的执行计划.
作用: 语句执行前,先看执行计划信息,可以有效的防止性能较差的语句带来的性能问题.
如果业务中出现了慢语句，我们也需要借助此命令进行语句的评估，分析优化方案。
(2) select 获取数据的方法
	1. 全表扫描(应当尽量避免,因为性能低)
	2. 索引扫描
	3. 获取不到数据
(3) 分析的是优化器按照内置的cost计算算法，最终选择后的执行计划。
	cost：代价、成本。对于计算机来讲就是消耗资源的多少？（主要cpu，IO，内存）
```



### 2、查看执行计划

```mysql
explain select * from world.city;
desc select * from world.city;
```

+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+


			执行计划显示结果分析

		range：索引范围查询 使用：>,<,<=,>=,like,in,or,between and
		ref：辅助索引等值查询 使用 =
		eq_ref：多表连接中，非驱动表连接条件是主键或唯一键 on后面用的是非驱动表的主键。on后 =两边的列字符类型要一样，不一样不走索引。
		const(system):聚簇索引等值查询

				 select * from city where id not in (10,20);

  key			此次查询最后选择的索引

	key_len 可以帮助我们判断，此次查询，走了联合索引的几部分。
	
	key_len的计算：	
		假设，某条查询（select * from t1 where a = and b= and c=）完全覆盖三列的联合索引，key_len= a长度+b长度+c长度？
			长度是列的最大处置字节长度：受数据类型，字符集影响
				数字：
					是否非空			not null		没有not null
					tinyint：1字节			1				1+1
					int：	4字节			4				4+1
					bigint：	8字节			8				8+1
				
				字符：utf8			一个字符最大占3个字节
									not null		没有not null
    				char（10）	 	 3*10			3*10+1
    				varchar（10）		 3*10+2			3*10+2+1
    				
    				utf8bm4			一个字符最大占3个字节
									not null		没有not null
    				char（10）	 	 4*10			4*10+1
    				varchar（10）		 4*10+2			4*10+2+1
    				
    			例子：
    			create table t1(
                a int not null,
                b int,
                c char(10) not null,
                d varchar(10)
                )charset = utf8mb4;
                
                 alter table t1 add index idx(a,b,c,d);
                 desc select * from t1 where a=1 and b=1 and c='a' and d='a'; 
                
                

5、rows 			此次查询需要扫描的行数
6、Extra			额外的信息

## 四、索引应用规范

### 1、说明

```bash
为了使索引的使用效率更高，在创建索引时，必须考虑在哪些字段上创建索引和创建什么类型的索引。那么索引设计原则又是怎样的?
```



### 2、（重点）建表时一定要有主键，一般是个无关列

```bash
1、建表一定要有主键，无关列
2、主键尽量是数字
3、尽量自己定制方便控制
```



### 3、选择唯一性的索引列

```bash
1.唯一性索引的值是唯一的，可以更快速的通过该索引来确定某条记录。
	例如：学生表中学号是具有唯一性的字段。为该字段建立唯一索引可以很快的确定某个学生的信息。如果使用姓名的话，可能存在同名现象，从而降低查询速度。
	优化方案：
	(1) 如果非得使用重复值较多的列作为查询条件(例如:男女),可以将表逻辑拆分
	(2) 可以将此列和其他的查询类,做联和索引
    select count(*) from world.city;
    select count(distinct countrycode) from world.city;
    select count(distinct countrycode,population ) from world.city;
```



### 4、（重要）为经常需要where、order by、group by、join on 等操作字段添加索引

```mysql
排序操作会浪费很多时间。
where  A B C      ----》 A  B  C
in 
where A   group by B  order by C
A,B，C

如果为其建立索引，优化查询
注：如果经常作为条件的列，重复值特别多，可以建立联合索引。
```



### 5、尽量使用前缀来索引

```mysql
如果索引字段的值很长，最好使用值的前缀来索引。
```



### 6、限制索引的数目

```mysql
索引的数目不是越多越好。
可能会产生的问题:
(1) 每个索引都需要占用磁盘空间，索引越多，需要的磁盘空间就越大。
(2) 修改表时，对索引的重构和更新很麻烦。越多的索引，会使更新表变得很浪费时间。
(3) 优化器的负担会很重,有可能会影响到优化器的选择.
percona-toolkit中有个工具,专门分析索引是否有用
```



### 7、删除不再使用或者很少使用的索引(percona toolkit)

```mysql
使用pt-duplicate-key-checker找到不再需要的索引

	表中的数据被大量更新，或者数据的使用方式被改变后，原有的一些索引可能不再需要。数据库管理员应当定期找出这些索引，将它们删除，从而减少索引对更新操作的影响。
```



### 8、大表加索引要在业务不繁忙期间操作

```mysql
白天用户访问多，业务量大。应该选择夜晚，访问量小的时候添加索引。减少服务器的压力
```



### 9、尽量少在经常更新值的列上建索引



### 10、建立索引的原则

```mysql
(1) 必须要有主键,如果没有可以做为主键条件的列,创建无关列
(2) 经常做为where条件列  order by  group by  join on, distinct 的条件(业务:产品功能+用户行为)
(3) 最好使用唯一值多的列作为索引,如果索引列重复值较多,可以考虑使用联合索引
(4) 列值长度较长的索引列,我们建议使用前缀索引.
(5) 降低索引条目,一方面不要创建没用索引,不常使用的索引清理,percona toolkit(xxxxx)
(6) 索引维护要避开业务繁忙期
```



## 五、不走索引的情况（开发规范）

### 1、没有查询条件，或者查询条件没有建立索引

```mysql
select * from tab;       全表扫描。
select  * from tab where 1=1;
在业务数据库中，特别是数据量比较大的表。
是没有全表扫描这种需求。
1、对用户查看是非常痛苦的。
2、对服务器来讲毁灭性的。
（1）
select * from tab;
SQL改写成以下语句：
select  * from  tab  order by  price  limit 10 ;    需要在price列上建立索引
（2）
select  * from  tab where name='zhangsan'          name列没有索引
改：
1、换成有索引的列作为查询条件
2、将name列建立索引

```



### 2、查询结果集是原表中大部分数据，应该是25%以上

```mysql
原因：
	查询的结果集，超过了总数行数25%，优化器觉得就没有必要走索引了。

解决办法：

	1、假如：tab表 id，name    id:1-100w  ，id列有(辅助)索引
select * from tab  where id>500000;
如果业务允许，可以使用limit控制。

	2、结合业务判断，有没有更好的方式。如果没有更好的改写方案，尽量不要在mysql存放这个数据了。放到redis里面。
```



### 3、索引本身失效，统计数据不真实

```mysql
原因：
1、索引有自我维护的能力。
2、对于表内容变化比较频繁的情况下，有可能会出现索引失效。

办法：
	一般是删除重建

现象:
有一条select语句平常查询时很快,突然有一天很慢,会是什么原因
select?  --->统计数据不真实,索引失效
DML ?   --->锁冲突

innodb_index_stats
innodb_table_stats
```



### 4、查询条件使用函数在索引列上，或者对索引列进行运算，运算包括(+，-，*，/，! 等)

```mysql
例子：
错误的例子：select * from test where id-1=9;
正确的例子：select * from test where id=10;
算术运算
函数运算
子查询
```

### 6、<> ，not in 不走索引（辅助索引）

```mysql
EXPLAIN  SELECT * FROM teltab WHERE telnum  <> '110';
EXPLAIN  SELECT * FROM teltab WHERE telnum  NOT IN ('110','119');

mysql> select * from tab where telnum <> '1555555';
+------+------+---------+
| id  | name | telnum  |
+------+------+---------+
|    1 | a    | 1333333 |
+------+------+---------+
1 row in set (0.00 sec)
mysql> explain select * from tab where telnum <> '1555555';

单独的>,<,in 有可能走，也有可能不走，和结果集有关，尽量结合业务添加limit
or或in  可以改成union，使用不同的条件，分别测试。
EXPLAIN  SELECT * FROM teltab WHERE telnum  IN ('110','119');
改写成：
EXPLAIN SELECT * FROM teltab WHERE telnum='110'
UNION ALL
SELECT * FROM teltab WHERE telnum='119'
```
